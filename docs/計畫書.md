# 台灣吸菸區地圖 TWsmokemap — 專案計畫書

> 版本 v1.0 ｜ 撰寫日期 2026-05-31 ｜ 專案類型：公益性響應式網站

---

## 一、專案背景與目標（Context）

台灣正推動「無菸城市」。現況痛點：

- **吸菸者**常因找不到合法吸菸區而違規吸菸（依菸害防制法，違規最高罰 1 萬元）。
- **非吸菸者**苦於不知何處易聞到煙味、也缺乏管道反映。
- 民眾普遍不清楚 112/03/22 修法後的禁菸／吸菸規定。

本專案建置一個 **響應式網站（電腦＋手機皆可用）**，以三個使用者目標為核心，依優先順序：

| 優先序 | 目標 | 服務對象 |
|--------|------|----------|
| 1（最高） | **找吸菸區** — 快速找到周圍合法／實際的吸菸區 | 吸菸者 |
| 2 | **回報煙味** — 標記「常聞到煙味」的地點，形成熱區 | 非吸菸者 |
| 3 | **宣導規則** — 簡明說明菸害防制法的禁菸／吸菸規定 | 全體民眾 |

**設計原則**：畫面乾淨簡潔、以使用者目標為導向、操作優先；盡量用免費資源；實體上架並同步 GitHub。

## 二、關鍵決策（已與使用者確認）

| 項目 | 選擇 | 理由 |
|------|------|------|
| 地圖服務 | **Google Maps Platform** | 使用者想學、有預算；每月 10,000 次地圖載入免費額度，小型公益站通常足夠 |
| 後端／資料庫 | **Supabase**（PostgreSQL + PostGIS） | 免費額度大、內建地理查詢與認證，最適合「附近搜尋」與眾包 |
| 部署平台 | **Vercel** | 連結 GitHub 自動部署、免費、設定最簡 |
| 上線範圍 | **台北市試點先行** | 開放資料最完整，流程跑順、驗證體驗後再全台擴張 |

> **重要事實（2025/03 起 Google 計費改制）**：取消舊的每月 $200 抵用金，改為「每個 SKU 每月免費額度」。地圖載入屬 Essentials，**每月 10,000 次免費**。但**仍須綁信用卡建立帳單帳戶**才能取得 API Key。本計畫含完整申請教學與「防爆量」設定，確保不會意外扣款。

### 資料來源調查結論

台灣**沒有**全國統一的「吸菸區」開放資料集；法規以「禁菸場所」為主，戶外吸菸區多由場所自設。但部分縣市／單位有開放資料，可作為試點基礎：

- [臺北市指定吸菸區](https://data.gov.tw/dataset/177504)（含行政區、地址、經緯度）— 本試點唯一採用的開放資料來源

→ 策略採 **「開放資料匯入 ＋ 使用者眾包」雙軌**，台北市先行。

## 三、技術堆疊

- **前端**：React 18 + Vite + TypeScript
- **樣式／RWD**：Tailwind CSS（行動優先、簡潔）
- **地圖**：`@vis.gl/react-google-maps`（Google 官方 React 套件）
- **後端／資料庫**：Supabase（Postgres + PostGIS、Auth、Row Level Security、JS Client）
- **部署**：Vercel（GitHub 連動自動部署）
- **版控**：GitHub（公開 repo，含 README、計畫書、開發紀錄）
- **測試**：Vitest（單元）、React Testing Library（元件）、Playwright（E2E）、Lighthouse / Web Vitals（效能）、k6（負載，選用）

## 四、系統架構

```
使用者瀏覽器 (RWD 網站)
   │  React + Vite (靜態前端，部署於 Vercel)
   │
   ├── Google Maps JS API ──► 地圖底圖、定位、標記、路線連結
   │
   └── Supabase JS Client ──► PostgREST / RPC
                               ├─ smoking_areas（吸菸區）
                               ├─ smell_reports（煙味回報）
                               └─ PostGIS 函式：nearby_smoking_areas()
```

- 前端為靜態站，敏感金鑰處理：Supabase 用 **anon public key + Row Level Security**（可安全放前端）；Google Maps key 透過 **HTTP referrer 限制 + API 限制 + 用量上限**保護。
- 不需自建伺服器，零維運成本。

## 五、資料模型（Supabase / PostGIS）

**table `smoking_areas`（吸菸區）**

| 欄位 | 型別 | 說明 |
|------|------|------|
| id | uuid (pk) | 主鍵 |
| name | text | 地點名稱 |
| address | text | 地址 |
| district | text | 行政區（如「信義區」） |
| location | geography(Point) | 經緯度（PostGIS） |
| kind | text | indoor / outdoor |
| source | text | official（開放資料）/ user（眾包） |
| status | text | approved / pending |
| note | text | 補充說明 |
| upvotes | int | 社群投票（驗證真實性） |
| created_at | timestamptz | 建立時間 |

**table `smell_reports`（煙味回報）**

| 欄位 | 型別 | 說明 |
|------|------|------|
| id | uuid (pk) | 主鍵 |
| location | geography(Point) | 回報座標 |
| report_type | int | 回報狀況：1＝我看到有人在抽菸、2＝我聞到濃烈煙味、3＝我聞到些微煙味 |
| time_slot | text | 早/中/晚/深夜 |
| note | text | 選填描述 |
| created_at | timestamptz | 建立時間 |

**RPC（PostGIS 附近搜尋）**：`nearby_smoking_areas(lat, lng, radius_m)` → 用 `ST_DWithin` 回傳半徑內、依距離排序的吸菸區。

**安全（RLS）**：所有人可 `select` approved 資料；`insert` 開放但寫入 status=pending（眾包）；`update/delete` 僅限服務端／管理者。煙味回報只允許 insert + 讀取彙總，降低濫用。

## 六、功能規格（依優先順序）

### 功能 1：找吸菸區（最高優先）
- 進站自動請求定位（可拒絕，預設台北市中心）。
- 地圖顯示附近吸菸區標記（官方資料 vs 眾包用不同顏色／圖示，眾包標「待驗證」徽章）。
- 「附近清單」側欄：依距離排序，顯示距離、室內外、地址。
- 點標記 → 詳情卡片：名稱、地址、來源、「以 Google Maps 導航」按鈕、「這裡正確嗎？」投票。
- 篩選：室內／室外、僅顯示官方驗證。

### 功能 2：回報煙味（次優先）
- 「回報煙味」按鈕 → 在地圖上拖曳放置圖釘或用目前定位。
- 表單：**回報狀況（三選一，描述性用詞）**、時段、選填描述 → 寫入 `smell_reports`。
  - ① 我看到有人在抽菸
  - ② 我聞到濃烈煙味
  - ③ 我聞到些微煙味
- 以**熱區圖（heatmap）**呈現聚合結果，而非逐筆精確點位（降低濫用、保護隱私）。
- 防濫用：每裝置／IP 速率限制（前端節流 + Supabase 端基本限制）。

### 功能 3：宣導吸菸規則（基礎優先）
- 靜態資訊頁：台灣菸害防制法重點（112/03/22 修法後）——
  - 禁菸場所、室外吸菸區設置限制（不得大於室外面積 1/2、不得設於必經之處）、新增禁菸品項、罰則（違規最高 1 萬元）。
  - 資料附官方來源連結（國健署 / 健康九九）。
- 簡潔卡片式版面、常見問答（FAQ）。

### 共用 UX
- 行動優先單欄、桌機雙欄（地圖＋清單）。
- 底部導覽（手機）：找吸菸區 / 回報煙味 / 規則。
- 載入骨架、空狀態、錯誤提示；繁體中文為主。

## 八、Google Maps API Key 申請教學（P0 執行）

1. 登入 <https://console.cloud.google.com> → 建立專案（如 `twsmokemap`）。
2. 左選單 **計費 Billing** → 建立帳單帳戶，綁定信用卡（必要步驟；低流量落在免費額度內不會扣款）。
3. **API 和服務 → 程式庫**：啟用「**Maps JavaScript API**」；若用地點搜尋再加「Places API」。
4. **憑證 Credentials → 建立憑證 → API 金鑰**。
5. **保護金鑰（重要，防盜用扣款）**：
   - 應用程式限制 → **HTTP 參照網址**：加入 `localhost:5173/*` 與正式網域 `https://你的網域/*`。
   - API 限制 → 僅勾選有啟用的 API。
6. **設用量上限防爆量**：Billing → 預算與快訊，設每月預算（如 NT$0–300）＋ Email 警示；必要時於 API 配額設每日上限。
7. 金鑰放 **`.env`（`VITE_GOOGLE_MAPS_API_KEY`）**，加入 `.gitignore`；Vercel 後台另設環境變數。**切勿把金鑰提交到 GitHub**。

### Supabase 設定步驟（P0–P1）

1. <https://supabase.com> 建立免費專案，記下 `Project URL` 與 `anon public key`。
2. SQL Editor 啟用 PostGIS：`create extension if not exists postgis;`
3. 建立上述 table、`nearby_smoking_areas` RPC、RLS policy。
4. 前端用 `@supabase/supabase-js` + `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`。

## 九、測試計畫

> **測試執行方式（含使用者體驗驗收）**：自動化測試（單元／元件／E2E）**由我（開發端）執行**並回報結果。對於**使用者體驗**這類需人眼判斷的部分，我會：
> 1. **能自動截圖時**：用 Playwright 自動產生「電腦版（1280px）」與「手機版（375px）」兩種畫面截圖，直接顯示給你檢查；
> 2. **無法在此環境執行時**：把可在你電腦本機跑起來的網頁（`npm run dev`）交給你，並附上各頁面在電腦版／手機版的預期畫面說明，由你親自開啟檢查並回饋。

### 功能測試（對應驗收條件）

| 功能 | 測項 | 工具 |
|------|------|------|
| 找吸菸區 | 定位成功/拒絕 fallback、附近半徑回傳正確排序、標記與清單一致、導航連結正確 | Playwright E2E + Vitest |
| 回報煙味 | 表單驗證、三選一回報狀況寫入成功、熱區更新、速率限制生效 | Playwright + RTL |
| 規則頁 | 內容渲染、外部連結可達、RWD 版面 | RTL |
| 通用 | PostGIS RPC 單元測試、RLS 權限（匿名不可改他人資料） | Vitest + Supabase 測試 |

### 使用者體驗（UX）驗收
- 每完成一個功能頁，產出**電腦版 + 手機版截圖**供你檢查版面、操作流程是否乾淨直覺。
- 重點檢查：首屏即懂、主要操作 ≤ 2 步、手機單手可操作、文字可讀、空/載入/錯誤狀態友善。
- 你的回饋直接納入修正，再進入下一階段。

### 效能測試
- **Lighthouse**（行動＋桌機）目標：Performance ≥ 90、Accessibility ≥ 90。
- **Core Web Vitals**：LCP < 2.5s、INP < 200ms、CLS < 0.1。
- **地圖載入**：延遲載入地圖、標記叢集（marker clustering）避免大量標記卡頓。
- **Google Maps 配額監控**：確認流量在免費額度內。
- （選用）**k6** 對 Supabase 附近搜尋做輕量負載測試。

### RWD／跨裝置測試
- 斷點：手機 (375px)、平板 (768px)、桌機 (1280px+)。
- 瀏覽器：Chrome、Safari (iOS)、Firefox。
- 真機驗證定位與觸控操作。

## 十一、風險管理

| 風險 | 影響 | 對策 |
|------|------|------|
| Google Maps 意外扣款 | 成本 | referrer 限制 + 預算警示 + 每日配額上限 |
| 眾包資料造假／濫用 | 資料品質 | 眾包標「待驗證」、社群投票、煙味改用熱區彙總、速率限制、Supabase 後台人工審核 |
| 開放資料不全（僅部分縣市） | 覆蓋率 | 試點台北先行 + 眾包補充，逐步擴張 |
| 金鑰外洩 | 安全 | `.env` + `.gitignore`，Vercel 環境變數，金鑰加限制 |
| 隱私（煙味回報含位置） | 法遵 | 不蒐個資、座標模糊化呈現、提供隱私說明 |
